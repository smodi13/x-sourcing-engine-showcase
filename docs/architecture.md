# Architecture

This document explains the full pipeline and data flow of the deterministic X
sourcing engine. The public site in this repository is a read-only replay of a
completed run. It reads sanitized static JSON and never contacts the X API.

## One-directional pipeline

```
Sourcing thesis
  to Query configuration
  to Count preflight
  to Recent search
  to URL resolution
  to Evidence classification
  to Builder attribution
  to Deduplication
  to Project consolidation
  to Selective profile enrichment
  to Diligence queue
  to Human investment decision
```

Data flows in one direction. Each stage is deterministic Python. No language
model assigns a score or a disposition at any stage.

## Stages

1. **Sourcing thesis.** A mandate is expressed as topics and discovery lanes.
2. **Query configuration.** Topics and lanes are combined into a small curated
   set of queries. Query text, fields, and request policy are versioned. See
   `config/pilot_queries.example.yaml` and `config/broad_queries.example.yaml`.
3. **Count preflight.** A counts-only request sizes seven-day volume before any
   retrieval, so spend is planned rather than discovered. See
   `src/engine/search/count_preflight.py`.
4. **Recent search.** The official recent-search endpoint returns posts over a
   seven-day window. A validator rejects unsupported operators before any call.
   See `src/engine/search/query_validator.py`.
5. **URL resolution.** URLs are extracted, expanded, and canonicalized. Links
   back to X are excluded, since a link to X is not an external artifact. See
   `src/engine/classification/url_resolution.py`.
6. **Evidence classification.** Each material claim is tagged Level A to D. See
   `src/engine/classification/evidence.py`.
7. **Builder attribution.** The engine decides whether the author is the
   builder, a third party, or a commentator. See
   `src/engine/classification/builder_attribution.py`.
8. **Deduplication.** Within-run and cross-run duplicates are removed, and
   multi-query matches are recorded. See `src/engine/consolidation/dedup.py`.
9. **Project consolidation.** Records are consolidated into companies or
   projects by normalized name. See `src/engine/consolidation/consolidate.py`.
10. **Selective profile enrichment.** Only shortlisted profiles are enriched.
    Follower count and verification do not determine advancement. See
    `src/engine/enrichment/profile_enrichment.py`.
11. **Diligence queue.** Actionable leads become an auditable queue with reason
    codes and unresolved questions.
12. **Human investment decision.** A person selects the company to advance. The
    engine organizes evidence and structures comparison. It does not decide.

## Governance overlay

Every paid request is estimated first, gated by an explicit approval bound to a
SHA-256 request fingerprint, protected by an execution lock, and tracked in
exact Decimal arithmetic against a fixed allowance. Raw responses are stored
before any derived output is written. See `docs/cost-controls.md`.

## Demo data flow

The replay site reads the JSON in `public/demo-data/`, generated by
`scripts/build_sanitized_demo_data.py` from the saved completed-run outputs. The
site renders entirely from that static data at build time.
